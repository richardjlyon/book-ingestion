# M2.1 — Full EPUB IR backend (`survey` + `extract_chapter` for EPUB)

> Design brief for the implementer who knows nothing about the wider system. Describes what M2.1 is for, what good looks like, and the design questions the brainstorm has to settle. Leaves *how* to the brainstorm.

Drafted 2026-05-14, post-Rabkin trial.

---

## 1. The bigger picture

`book-ingestion` is the parsing half of a two-tool split. The other half is the `ingesting-books` skill in Richard's Obsidian vault, which takes the JSON IR this tool produces and authors structured vault content (notes, concepts, threads, source files) chapter-by-chapter under a strict authoring contract.

The pattern was validated end-to-end on 2026-05-14 against Rabkin's *What is Modern Israel?* (PDF, 9 chapters, ~190 atomic notes, 70 people files, 8 new threads, 3 grouped MOC promotions). The skill works. The tool works. The two compose cleanly.

The next book in Richard's queue is Pappé's *The Ethnic Cleansing of Palestine* (2007, Oneworld Publications). Pappé is **EPUB-only** in his collection. The skill expects `book-ingest survey <path>` and `book-ingest extract --chapter N <path>` to work against it. Today they don't — M2.0 ships EPUB *metadata* extraction (sub-second, via OPF parsing) but not the structural IR backend that `survey` and `extract_chapter` require.

This milestone closes that gap. After M2.1, the `ingesting-books` skill runs against any EPUB the same way it runs against any PDF, and the `book-ingest` CLI's surface is symmetric across both formats.

---

## 2. What gets produced when an EPUB is ingested

The end state, after M2.1 ships:

### 2.1 `book-ingest survey <epub-path>` returns a `BookSurvey`

The same `BookSurvey` JSON shape the PDF path returns today. Specifically:

- A `map.chapters` list with one entry per printed chapter, each carrying `index`, `title`, `start_page`, `end_page` (or the EPUB equivalent — see §5.1 below), and `provenance` (`embedded` / `inferred` / `none`).
- A `map.provenance` field at the document level (`embedded` if derived from the EPUB's own navigation; `inferred` if the brainstorm decides typographic / structural inference is needed; `none` if the structure couldn't be recovered).
- `quality.flags` from the same closed vocabulary the PDF path uses, plus any EPUB-specific flags the brainstorm needs to add.

The skill's Step 0 reconciliation procedure (compare survey output to the printed Contents page) should work against EPUB output in the same shape it works against PDF output.

### 2.2 `book-ingest extract --chapter N <epub-path>` returns a `ChapterContent`

The same `ChapterContent` JSON shape the PDF path returns today. Specifically:

- `simple_view`: a flat list of typed blocks (`paragraph`, `heading`, `table`, `figure_caption`, `footnote`, `page_break`, `failed_region`) in reading order.
- Each block carries `page` (the EPUB-equivalent of PDF page index — see §5.1) and `page_label` (the printed-page-equivalent — also §5.1).
- Quality flags per the closed vocabulary.

The skill's Step 4 (atomic note extraction with `cite_locator`) should be able to produce locators of shape `"chN, p X"` against EPUB-derived block content the same way it does against PDF-derived content.

### 2.3 Symmetry with the existing PDF backend

The `BookSurvey` and `ChapterContent` types are unchanged at the public schema level. The skill consumes EPUB output and PDF output indistinguishably. The differences live inside the tool — different extractors, different parsing libraries, different failure modes — but never leak through the schema.

The cache, the CLI flags, the `--no-cache` and `--cache-dir` semantics, the exit-code contract (0 / 2 / 1) all behave identically.

## 3. What this milestone has to output to make that possible

Three outputs make the wider system work after M2.1.

### 3.1 An EPUB extractor that produces the IR shape

A new extractor module under `src/book_ingestion/extractors/` (or wherever the brainstorm decides — the existing `extractors/epub.py` handles M2.0 metadata; M2.1 likely extends or sits alongside). The extractor parses the EPUB structure (OPF + spine + nav/NCX + content XHTML) and produces the same `BookSurvey` and `ChapterContent` types the PDF path produces today.

### 3.2 An honest answer to the page_label question

EPUBs are reflowable. There are no fixed pages. The skill's `cite_locator` convention assumes a printed-page number is available. M2.1 has to decide what `page_label` means for EPUB and document the decision. See §5.1 for the design question.

### 3.3 An IR that survives Pappé end-to-end

The acceptance fixture: a clean `book-ingest survey pappé.epub` plus `book-ingest extract --chapter N` invocations across all of Pappé's chapters, producing IR that the `ingesting-books` skill can consume without skill-side workarounds. If the skill has to invent procedures the PDF path doesn't require, the IR is leaking the format-specific differences and M2.1 is incomplete.

## 4. Components the brainstorm should consider

### 4.1 Library choice

The Python EPUB ecosystem has three reasonable options:

- **`ebooklib`** — the de-facto Python EPUB library. Reads OPF, spine, nav, NCX, and content XHTML. Mature; some quirks. ~3 MB install footprint.
- **Direct `zipfile` + `lxml` parsing** — EPUB is just a ZIP with XML inside. Roll your own. More control, more work, fewer dependencies (lxml is already broadly available).
- **`pypub` / others** — less mature, less maintained.

The M2.0 EPUB metadata extractor uses stdlib `zipfile` + the OPF parser the brainstorm chose at the time. M2.1 should consider whether to follow the same pattern (consistency) or adopt `ebooklib` for the heavier structural work (less bespoke code).

### 4.2 Chapter detection

Three sources of chapter structure inside an EPUB:

- **OPF spine** — the canonical reading order. One spine item per content file. Maps to printed chapters with varying fidelity (one-file-per-chapter is common; multi-file chapters and chapter-spanning files both exist).
- **nav.xhtml** (EPUB 3) or **NCX** (EPUB 2) — the navigation table the reader sees. Hierarchical; closer to the printed Contents page.
- **Heading structure inside content files** — `<h1>`, `<h2>` inside the XHTML.

Each is a different signal of where chapters break. The brainstorm has to decide the precedence order and the reconciliation logic when the three disagree (analogous to the PDF case where the embedded outline disagrees with the printed Contents page — the recipe Rabkin's ingest validated for PDF will need an EPUB equivalent).

### 4.3 XHTML cleanup

EPUBs vary wildly in XHTML quality. Some are cleanly authored with semantic markup; some are reflowed PDF dumps with inline styles, broken structure, garbage attributes. The extractor needs a cleanup pipeline that strips chrome (navigation links, footer text, page-number remnants from print conversions) and produces the typed blocks the IR expects.

The `defuddle` tool (kepano/obsidian-skills, NPM `defuddle-cli`) does similar cleanup for web pages and is now installed in Richard's Claude Code skills. The brainstorm should look at whether the same approach (or even the same library, if it's invocable from Python) fits the EPUB case.

### 4.4 Quality scoring

The PDF backend uses Docling's per-page confidence scores to derive `EXCELLENT` / `GOOD` / `FAIR` / `POOR` per block. EPUB has no equivalent — there's no parser confidence in zip + XML. The brainstorm has to decide what `confidence` means for EPUB blocks. Three options:

- Always `EXCELLENT` for EPUB content (the parsing is deterministic; if you got the bytes, the bytes are right).
- Heuristic from XHTML cleanliness — count broken tags, missing closing elements, suspicious patterns.
- Drop `confidence` for EPUB and signal quality only via `flags`.

### 4.5 Integration with M2.0

The M2.0 EPUB metadata extractor already opens the EPUB and parses the OPF. M2.1 will do the same. The brainstorm should decide whether M2.1 reuses M2.0's parsing (shared infrastructure, single source of truth for OPF) or duplicates it (cleaner separation, no risk of M2.1 breaking M2.0).

### 4.6 Cache integration

The existing cache is content-hash-keyed and survives moves. EPUB hashing is identical to PDF hashing (just `sha256` the bytes). M2.1 should integrate with the existing cache the same way the PDF path does, with a cache-version bump if the IR shape changes for EPUB-specific reasons.

## 5. Open design questions for the brainstorm to resolve

These are the questions the brainstorm needs an answer for before writing code. They're not answered in this brief because the right answer depends on engineering trade-offs the implementer is best positioned to weigh.

### 5.1 What does `page_label` mean for EPUB?

EPUBs have no fixed pages. The PDF backend gives every block a `page` (PDF page index) and a `page_label` (the printed page number, where recoverable from the PDF's `/PageLabels` stream or inferred). The skill's `cite_locator` convention writes `"ch3, p 142"` using `page_label`.

For EPUB, three plausible answers:

- **(a) Synthesise pseudo-pages from word count.** Pick a word count per pseudo-page (e.g. 250 words = 1 pseudo-page) and number from the start of the chapter. Locators become `"ch3, pseudo-p 5"` or `"ch3, p 5"` with the pseudo-page convention documented in source-file Provenance. Pro: numerical-looking; familiar shape. Con: synthetic; not stable across re-extraction if word count changes; may not align with anything a reader can find.

- **(b) Use OPF spine index plus offset within item.** Each block carries `(spine_idx, block_idx)` as its location. Locators become `"ch3, item 4 / block 12"`. Pro: deterministic, stable across re-extraction. Con: opaque to a reader holding the EPUB in their reader app; doesn't map to anything the reader sees.

- **(c) Use chapter-relative percentage or block-index.** `"ch3, 38% in"` or `"ch3, block 12"`. Pro: simple; chapter-scoped. Con: same opacity as (b); percentage-in shifts if extraction changes.

- **(d) Drop `page_label` for EPUB and let `cite_locator` use chapter-only granularity.** `"ch3"` is the locator; pages don't exist. Pro: honest about the medium. Con: loses the per-passage anchoring that's the point of `cite_locator` for books.

Each option has different implications for skill-side code (the existing `cite_locator` rules in `ingesting-books/SKILL.md` assume a numeric page). The brainstorm should pick one, document the choice, and note the skill-side adjustment needed (which Cowork will land in `ingesting-books/SKILL.md`).

The brainstorm may also propose a hybrid: if the EPUB *does* carry printed-page markers (some EPUBs derived from print editions include `<a class="page" id="page-142"/>` anchors), use those when present and fall back to one of (a)-(d) when not.

### 5.2 How to detect printed-chapter boundaries when OPF / nav / heading structure disagree

The PDF case has the survey-vs-printed-TOC reconciliation recipe (Rabkin trial, validated across 6 structural shapes). EPUB needs an analogous recipe.

The brainstorm should think through:

- One-file-per-chapter EPUBs (the easy case): spine and nav agree; chapters map 1:1 to spine items.
- Multi-file-per-chapter EPUBs: one printed chapter spans N spine items. Combining heuristic needed.
- Chapter-spanning-file EPUBs: one spine item contains N printed chapters separated by `<h1>` or `<h2>` inside. Splitting heuristic needed.
- EPUBs with broken nav: spine present, nav malformed or absent. Fall back to spine-only.
- EPUBs with broken spine (extreme): synthesised reading order from heading structure inside content files.

The recipe should produce the same `map.chapters` shape regardless of which path was taken. The `map.provenance` field surfaces which (so the skill can spot-check at Step 0).

### 5.3 What goes in `simple_view` for EPUB extraction

PDFs produce `paragraph`, `heading`, `table`, `figure_caption`, `footnote`, `page_break`, `failed_region` blocks. EPUBs may produce additional types (e.g. `pre` / code blocks for technical books, `blockquote` blocks distinct from paragraphs). The brainstorm should decide whether to extend the `BlockType` enum or normalise everything into the existing types.

### 5.4 Handling DRM / encryption

M2.0 already detects DRM-protected EPUBs and fails cleanly with a structured error. M2.1's behaviour: same detection, same exit, no attempt at extraction. The brainstorm doesn't need to design new detection — just ensure M2.1 honours M2.0's existing path.

### 5.5 Cache schema version

If the IR shape changes for EPUB (new flags, new block types, new metadata fields), bump the cache version so existing PDF-cached entries don't get confused. The brainstorm decides the cache-version bump strategy.

## 6. Out of scope

The brainstorm should not extend M2.1's scope to cover these — they're reserved for later milestones or other tools.

- **OCR for scanned PDFs (M3).** Different problem, different milestone. M2.1 is purely about EPUB.
- **MOBI / AZW3 / KF8.** Amazon's Kindle formats. Calibre's `ebook-meta` is the natural future hook; M2.1 is EPUB-only.
- **DRM removal.** Detect-and-fail is the policy. The tool does not crack DRM under any circumstances.
- **`adding-references` skill integration changes.** That's a different skill, owned by the `ingesting-references` workflow. M2.0's `extract_metadata` already handles EPUB metadata for the adding-references case; M2.1 doesn't touch that surface.
- **EPUB authoring or modification.** This is a read-only tool. We extract; we don't write.
- **Skill-side adjustments to `ingesting-books/SKILL.md`.** Those land at the Cowork side once M2.1's design choices crystallise. The brainstorm should produce a note for the Cowork-side handoff describing what skill changes its design implies.

## 7. Acceptance — what good looks like

M2.1 has done its job when, given Pappé's *The Ethnic Cleansing of Palestine* EPUB:

1. `book-ingest survey pappé.epub` returns a clean `BookSurvey` with `map.provenance` set appropriately and `map.chapters` matching (or close enough to reconcile against) the printed Contents page.
2. `book-ingest extract --chapter N pappé.epub` for each of Pappé's chapters returns clean `ChapterContent` with no `failed_region` blocks under normal conditions.
3. The `ingesting-books` skill, run against Pappé under the validated channel pattern, produces vault content of the same quality and shape it produced for Rabkin — without skill-side code paths that exist purely to compensate for EPUB-vs-PDF differences.
4. Acceptance fixtures: at least two real-EPUB tests pinned in `tests/test_survey_real.py` and `tests/test_extract_real.py` (Pappé as the primary fixture; one further EPUB the implementer chooses for shape diversity).
5. The CLI's exit-code contract (0 / 2 / 1) behaves identically for EPUB as for PDF. The cache works for EPUB the same way it works for PDF.
